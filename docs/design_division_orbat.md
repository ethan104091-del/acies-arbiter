## 0. 設計理念（user 定）

1. **把師的編制完整、明確攤開** —— 玩家看得到師裡每個營級積木。
2. **最高自由度** —— 任何建制營都能單獨拉出來。想單拉一個砲兵營去送死？可以。想自組 RCT（步兵營+砲兵營+工兵）？自己配。
3. **裁判只管「編制真實 + 物理可行」，不管「戰術合不合理」** —— 你能拉出的營必須真實存在、走得到；蠢不蠢是你的自由（真實指揮官也能做蠢決定）。

**最小積木 = 營**（rules_v2 真實指揮的自然單位）。連太細(80+/師)、團太粗(拉不出單一砲營)。一個師約 15-18 個營，棋子數可控。

---

## 1. 人數模型（雙軌）

每個單位同時有：
- **`personnel`（人數）** —— 主要顯示，給你實感。拆分/歸建按人數加減，守恆。
- **`strength`（0-100 戰力%）** —— 戰鬥公式用（沿用現有零機率 CP 公式）。代表「該單位戰力完整度」，不是人數。

拉出一個營：母師 personnel 減該營人數、strength 按貢獻度微調（不是線性，砲兵走了步兵師還能打）。歸建相反。

面板顯示範例：
```
1ID  主力 ~9,500人  戰力92%
 └1-16營  870人  步兵  戰力90%
 └第5砲營 500人  105mm×12  戰力95%
```

## 3. 操作:拆 / 編組 / 歸建

**拆（detach）**：把任一營從師拉出成獨立棋子。
```
「1ID 第5榴砲營 獨立到 (8,11)」→ 砲營變獨立棋子,1ID 扣 500人
```

**編組（group）**：把多個營綁成一個合成戰鬥群(自組 RCT/CCA)。
```
「1ID 組 RCT-16 = 1-16營 + 第5砲營 + 工兵營」
→ 三個營合成一個棋子,一起移動/戰鬥,戰力合計
```
- 合成棋子的兵種 = 主成分(有步兵→步兵群;純砲→砲群)
- 戰鬥時各兵種貢獻不同(砲營提供火力、步兵提供佔領/防禦)

**歸建（rejoin）**：營/戰鬥群併回母師,人數加回。

**完全自由**：以上組合裁判不阻止(只要營存在、走得到)。蠢決定(單砲營送死、步兵無砲硬攻城)照樣執行,後果自負。

## 4. state schema

師單位加 `orbat`（編制樹,列出所有營級積木 + 狀態）：
```json
"1ID": {
  ...現有師欄位...,
  "personnel": 13800,
  "orbat": {
    "1-16": {"name":"1營/16團","type":"infantry","personnel":870,"status":"in_division"},
    "5arty": {"name":"第5榴砲營","type":"artillery","personnel":500,"weapons":"105mm×12","status":"in_division"},
    ... 所有營 ...
  }
}
```
- `status`: in_division(在師裡) / detached(已拉出) / grouped:<群id>(編入某戰鬥群)
- 拉出的營/戰鬥群在 state.units 成獨立棋子,uid 如 `1ID-5arty`、`1ID-RCT16`

## 6. 跟現有系統的關係

- **取代 `detachments.py`**：舊的「各兵種固定比例拆」改成「營級積木自由拉/組」。
- **迷霧/視角**：拉出的營/戰鬥群是獨立棋子,自動套用 filter_state_for(敵方只看偵獲的)。
- **戰鬥公式**：營級單位用同一套 CP 公式,只是 strength 規模較小。
- **map.py / commander.py / hour 級命令 / 參謀**：都自動支援新棋子(它們在 state.units 裡)。
